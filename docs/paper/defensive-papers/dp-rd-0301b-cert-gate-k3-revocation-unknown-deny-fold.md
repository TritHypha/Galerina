> **Purpose.** Defensive publication; **novelty disclaimed** (§3) — most pointedly, Kleene three-valued logic and the generic "a `min`-fold collapses on any non-ALLOW operand" property are explicitly **not** claimed anywhere in this disclosure (Galerina's own paper index pre-grades "K3 governance gate as novel" as not publishable). What is recorded is a specific, narrower combination: TLS/PKI channel trust is decomposed into four *named* sub-verdicts (pinned-digest match, library chain-validation outcome, validity-window membership, revocation status) via `toSubVerdicts`, each independently mapped so that every missing, errored, timed-out, or otherwise un-provable factor defaults to INDETERMINATE and *never* to ALLOW; the revocation factor additionally requires provable *freshness* for a positive outcome (a stale or future-dated "good" degrades to the same INDETERMINATE as no response at all); the four fold by simple conjunction (`min`), so one unresolved factor denies the whole channel regardless of how healthy the other three are; and the resulting denial is distinguished in an audit diagnostic from an ordinary definite-negative denial. The problem this closes is named and real: the public web's documented practice of *soft-failing* an unreachable revocation check (treating "unknown" as "not revoked").

## 1. Technical field

TLS/PKI channel-admission gating; certificate pinning, chain-validation, expiry, and revocation-status composition as a governance-layer fold over the outputs of an existing TLS/PKI library; three-valued (Kleene K3) verdict composition applied specifically to the certificate/channel-trust boundary; defences against revocation-check soft-fail (CWE-299).

## 2. Background & problem

Standard TLS clients commonly treat "revocation status could not be determined" (OCSP responder unreachable or timed out, no stapled response, CRL fetch failed) as equivalent to "not revoked," i.e. they **allow** the connection. This *soft-fail* posture is publicly documented browser-vendor practice, and its own stated rationale is a real trade-off: a network attacker who can simply **block** the revocation check achieves the same effect as forging a "good" response, so refusing to connect on responder-unreachable was judged to trade availability for a protection that a capable network attacker could defeat anyway. The practical consequence, independent of that rationale's merits, is that online revocation checking provides little protection against exactly the network-position attacker it exists to catch (CWE-299, Improper Check for Certificate Revocation).

A second, compounding problem is structural rather than protocol-specific: when channel trust is decomposed into several independently-evaluated factors (is the presented leaf pinned; did the library's path validation succeed; is the cert inside its validity window; is it un-revoked), a soft-fail habit in *any single factor's* implementation quietly degrades the whole decision — three correctly-evaluated positive factors provide no protection if the fourth silently treats "I don't know" as "yes." `cert-gate.ts`'s own header names this precisely: *"Mapping a timed-out or errored factor to ALLOW is exactly the web's soft-fail mistake — it is encoded once here and tested for each of the four factors."*

## 4. Summary of the disclosed subject matter

A pure governance module (`cert-gate.ts`, "TLSTP S1") that takes the *outputs* of crypto/TLS validation already performed by a TLS/PKI library (never performing ASN.1/path-building/signature/OCSP parsing itself) and: (a) maps four named factors — `pinMatch`, `chainValid`, `notExpired`, `revocationFresh` — to Kleene-K3 trits via `toSubVerdicts`, with every missing/errored/un-provable input defaulting to `INDETERMINATE` (`0`), never `ALLOW` (`+1`); (b) folds the four via the shipped conjunctive `allOf` (= `vAnd`-reduce = numeric `min`) into one `certVerdict`, exactly `ALLOW` iff all four are `ALLOW`, a single `DENY` annihilating regardless of the others, and a single `INDETERMINATE` (with no `DENY` present) also denying; (c) optionally folds in zero-or-more degrade-only `sideSignals` via `withSideSignal` (No-Coercion: `vAnd(t*, r) ≤ t*` for all nine trit combinations — can only lower, never raise); and (d) collapses the result at the trust boundary via `decideAtBoundary` into a `BoundaryDecision`, structurally emitting the `FUNGI-GOV-3VL-001` diagnostic whenever an `INDETERMINATE` was the cause of the denial (impossible to drop silently), while an ordinary definite `DENY` (a confirmed pin mismatch or a confirmed-revoked cert) collapses to the same refused connection *without* that diagnostic. A companion pure helper, `revocationRecheckDue`, governs only the *cadence* at which a long-lived stream re-runs this fold mid-session, defaulting to a re-check at every chunk/frame boundary and degrading any absent, invalid, or unrecognized cadence configuration to that same floor — there is deliberately no configuration value meaning "never."

## 5. Detailed description

### 5.1 The four sub-verdicts and their fail-closed mappers
- **`pinMatchVerdict`:** a pinned digest present and matching the presented leaf digest → `ALLOW`; a pinned digest present and *mismatching* → `DENY` (a hard annihilator — "the MITM-with-valid-cert case"); no pin configured, or nothing presented to compare → `INDETERMINATE`. The module's own comment names the gotcha this avoids: *"pin mismatch is −1; pin absent is 0. They are different — conflating them either bricks un-pinned hosts (−1) or weakens pinned ones (+1)."*
- **`chainValidVerdict`:** the TLS library's own path-validation outcome — `"valid"` → `ALLOW`; `"invalid"` → `DENY`; `"incomplete"` or absent → `INDETERMINATE`.
- **`notExpiredVerdict`:** `now` inside `[notBefore, notAfter]` inclusive → `ALLOW`; outside → `DENY`; any of the three timestamps absent or non-finite → `INDETERMINATE` (a missing clock is treated as "cannot prove," not "assume valid").
- **`revocationVerdict`** (the headline factor): `"revoked"` → `DENY`; a `revocationCheck` callback that *throws* → `INDETERMINATE` (mirroring a specific, named sibling site, `fuse-loader.ts:537`); anything other than `"good"` (including `"unknown"` or absent) → `INDETERMINATE`. Critically, a `"good"` outcome must **also** be provably fresh — `revocationProducedAt`, `now`, and `revocationFreshnessMs` all present and finite, with `0 ≤ age ≤ freshnessMs` — or it too degrades to `INDETERMINATE`. A stale "good" (older than the freshness window) and a future-dated "good" (clock skew or replay) are both treated identically to no response.

### 5.2 The fold and the audited boundary collapse
`certVerdict = allOf([pinMatch, chainValid, notExpired, revocationFresh]) = min` of the four trits. `ALLOW` iff all four are `ALLOW`; correctness over the *entire* input space, not a sample, is machine-checked by an exhaustive 81-row (3⁴) truth table asserting `certVerdict === Math.min(...)` and `authorized ⟺ all four === ALLOW`, run both at the sub-verdict level and end-to-end through `certGate`. The headline property, worked in a named test: three `ALLOW` factors (valid pinned chain, in-window) cannot rescue a fourth `revocationFresh = INDETERMINATE` (responder unreachable) — the channel denies, and `decision.diagnostic?.code === "FUNGI-GOV-3VL-001"`. A definite `DENY` (e.g. a hash-pin mismatch on an otherwise library-valid chain) also denies, but with `diagnostic === null` — *"a definite −1 is an ordinary policy denial, not a collapsed 0"* — so an auditor can distinguish "we know this is bad" from "we don't know" while both refuse the connection.

### 5.3 Mid-stream re-check cadence — no encoding for "never"
A long-lived governed stream cannot rely on its opening revocation check indefinitely (a signing key can be revoked mid-session). `revocationRecheckDue(cadence, state)` answers only *when* a re-check is due — it never decides *whether* to allow (that stays entirely in §5.2's fold), so it structurally cannot weaken unknown→DENY; at most it forces more re-checks, never fewer. `DEFAULT_REVOCATION_CADENCE` is `{ mode: "chunk_boundary" }`. Fail-closed totality, test-verified: an absent, `null`, non-object, or unrecognized-`mode` configuration all degrade to the chunk-boundary floor; a `"poll"` mode with a non-finite or `≤0 everyMs` also degrades to the floor; a non-finite `msSinceLastCheck` is treated as due (a broken clock must not skip a re-check). There is no cadence value that means "never."

### 5.4 Degrade-only side-signal composition — reused, not new
`withSideSignal(verdict, sideSignal) = vAnd(verdict, sideSignal)`. By the same No-Coercion property disclosed generally in DP-RD-0270 (`min(digital, physical) ≤ digital`), a side-signal (e.g. a tamper/substrate-integrity reading) can sink an otherwise-`ALLOW` channel but can never manufacture an `ALLOW` from a lower verdict — verified here over all nine `(t*, r)` trit combinations, and end-to-end: a `DENY` side-signal sinks a healthy channel; an `INDETERMINATE` side-signal degrades and audits (`FUNGI-GOV-3VL-001`); an `ALLOW` side-signal changes nothing. This sub-mechanism is not itself claimed — it is the shipped generic rule, applied to this fold's output.

## 6. Honest limitations & scope

- **This is a fold over reported outcomes, not a validator.** `cert-gate.ts` performs no ASN.1 parsing, path-building, signature verification, or OCSP/CRL wire-protocol handling. A TLS/PKI library that *wrongly* reports `chainOutcome: "valid"` for an invalid chain passes straight through this fold unchanged — this construction can refuse to upgrade an honest "unknown" to "allow," but it cannot detect a confidently-wrong "allow" from its inputs.
- **Freshness is only as trustworthy as the supplied clock.** `now` and `revocationProducedAt` are caller-supplied; a caller with a broken or attacker-influenceable clock could cause a stale response to read as fresh, or vice versa. Clock integrity is assumed, not constructed, by this disclosure.
- **A `revocationCheck` that swallows its own errors defeats the throw→INDETERMINATE mapping at a layer this module cannot see.** §5.1's throw-handling is correct for an honestly-raised exception; a check implementation that catches its own failure and fabricates a `"good"` result is indistinguishable from a genuine one from inside `cert-gate.ts`.
- **Pin provisioning and rotation are assumed, not constructed.** `pinnedDigests` supply and lifecycle (a common operational cause of pinning outages) is out of scope.
- **The cadence helper reports; it does not enforce.** `revocationRecheckDue` only answers whether a re-check is due — a caller that never asks, or asks and discards the answer, reopens the opening-check-only gap §5.3 exists to close. This is the same "the gate exists but may go uncalled" caveat DP-RD-0269 already names generally (its `FO-GATE-UNCALLED` class); this disclosure does not add a call-graph proof that every long-lived stream in this codebase actually polls it on every chunk boundary.
- **No-Coercion is one-directional by design, which is also a real availability cost.** A noisy or miscalibrated side-signal cannot manufacture a false `ALLOW`, but it *can* wrongly sink a genuinely healthy channel — an availability risk this construction accepts by design, not a security gap, and the dual of the point DP-RD-0270 makes generally.
- **The empty-collection fold-identity pin (DP-RD-0272 §5.4) does not arise here and is not claimed.** `certVerdict` always folds exactly four fixed sub-verdicts, never a variable-length or possibly-empty collection, so the vacuous-truth failure mode that pin addresses is structurally absent from this construction, not solved by it.
- **Not claimed:** that this construction detects a certificate the underlying library itself misreports; that it replaces OCSP/CRL infrastructure; that fail-closed means unbreakable — an attacker able to force every factor to `INDETERMINATE` (e.g. by blocking all four checks) achieves a denial-of-service against the channel, which is the accepted, honest trade: a refused connection, never a silently-accepted bad one.

## 7. Illustrative disclosure claims (prior-art disclosures, not patent claims)

1. **A method** of gating a TLS/PKI channel-admission decision wherein the outputs of certificate-pin comparison, chain-path validation, validity-window membership, and revocation-status checking are each independently mapped to one of three verdict states — positive, negative, or indeterminate — such that every missing, errored, timed-out, or otherwise un-provable input maps to the indeterminate state and never to the positive state.
2. **A method** as in claim 1 wherein the four verdicts are combined by a conjunctive minimum such that the channel is admitted if and only if all four are positive, a single negative verdict overrides to refusal independent of the other three, and a single indeterminate verdict, in the absence of any negative, also results in refusal.
3. **A method** as in claims 1–2 wherein a positive revocation-status outcome is additionally required to fall within a bounded freshness window relative to the current time, such that a positive outcome older than the window, or dated after the current time, is remapped to the indeterminate state rather than treated as positive.
4. **A method** as in claims 1–3 wherein a refusal arising from an indeterminate verdict is distinguished, in an emitted diagnostic record, from a refusal arising from a definite negative verdict, so that a "status unknown" refusal and a "confirmed bad" refusal are separately auditable while both refuse the connection.
5. **A method** as in claims 1–4 wherein a pin-configured-but-uncomparable state (no digest presented to compare) is mapped to the indeterminate state while a pin-configured-and-mismatched state is mapped to the negative state, the two remaining distinct non-positive outcomes.
6. **A system** applying claims 1–5, further comprising a cadence function, independent of the admission fold, that determines whether a long-lived channel is due for a repeat evaluation of claims 1–4, wherein an absent, invalid, or unrecognized cadence configuration defaults to the most frequent supported re-check interval and no configuration value suppresses re-checking entirely.
