# Reference — Epilogue Receipts (the audit-receipt strategies)

The proof-of-execution receipt every governed flow emits, and the strategies that determine its assurance level.
This is the detail page behind the "Audit receipts" row in
[contract-authoring-model.md](../contract-authoring-model.md) and §6 of
[governance-automated-floor.md](../governance-automated-floor.md). Receipts are a **Table 2/3 (automated)** element:
you do **not** write the strategy in the contract — it is **profile-driven** — and a receipt is emitted regardless.

## A. What an Epilogue Receipt is

The final piece of the audit trail. After every governed flow execution a receipt is generated that proves four
things:

1. **What was executed** — `inputSeal` = `sha256` of source + contract hash.
2. **What was produced** — `outputSeal` = `sha256` of the result (set post-execution).
3. **When it ran** — `generatedAt` (ISO 8601 UTC).
4. **Under what governance** — a reference to the proof obligation.

It is generated by a `governed floor_3 flow` (a Table-3 floor operation) declaring only `audit.write`, and is
**emitted regardless of the contract** — the strategy chooses the *assurance*, not *whether* a receipt exists.

### Receipt fields

| Field | Meaning |
|---|---|
| `strategy` | which of the four strategies produced this receipt |
| `inputSeal` | `sha256:<hex>` of source + contract hash |
| `outputSeal` | `sha256:<hex>` of the execution result |
| `receiptId` | unique receipt identifier (`flowId:generatedAt`) |
| `flowId` | which flow the receipt covers |
| `generatedAt` | ISO 8601 UTC timestamp |
| `onFailure` | what happens if the receipt cannot be produced/verified (below) |
| `verified` | has DSS verified this receipt against the signed audit log? |

## B. The four strategies

#### `sha256_seal`  · [implemented]
**What** — a cryptographic **hash commitment**: `sha256` of source + contract (input) and of the result (output).
**Assurance / cost** — fast, always available. Proves *what ran and what it produced* without revealing the payload.
**Status** — implemented (Stage-A TypeScript mirror computes the seal).
**Result** — the default, real receipt. `auto` resolves to it.

#### `zk_snark_receipt`  · [stub]
**What** — a **zero-knowledge proof** receipt: proves the governed execution happened correctly without revealing
inputs/outputs.
**Assurance / cost** — highest assurance, slow.
**Status** — **explicit stub** — the prover backend is not integrated. Admissible in a **certified** profile only
with a real, non-placeholder circuit (never a placeholder).
**Result** — reserved for the highest-assurance tier once a real circuit lands; today it must not be relied on.

#### `auto`  · [resolves to `sha256_seal`]
**What** — let the compiler select the strategy based on the **ValueGraph risk tier**.
**Result** — in practice resolves to `sha256_seal` (the implemented strategy); the hook exists to escalate to a
stronger receipt automatically when a risk tier warrants and the backend exists.

#### `none`  · [opt-out]
**What** — no receipt — an explicit opt-out.
**Result** — permitted only in a **non-certified** profile; a certified profile mandates a real receipt (this is a
fail-closed floor, not a per-flow convenience).

## C. `onFailure` — what happens when a receipt can't be produced/verified

| Mode | Behaviour |
|---|---|
| `halt_pipeline` | stop the pipeline — the strictest, fail-closed choice |
| `quarantine_payload` | isolate the payload rather than release it |
| `log_and_continue` | record the failure and proceed (the current default in the prototype) |

## D. Where it sits

Every governed execution emits a receipt, and every security trap additionally appends an `AuditEvent` (CBOR Tag
410) to an **append-only** log. Verification (`verifyReceipt`) checks a receipt against that signed audit log — Stage
A simulates it (returns true for a well-formed receipt); Stage B is the real `DSS.wasm` deterministic-isolation TCB
(#102–106, post-P9). See [governance-automated-floor.md](../governance-automated-floor.md) §6.
