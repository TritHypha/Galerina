# sink-canonicality-allowlist — sink-SHAPED stdlib entries that are NOT an
# untracked egress/exec/write sink (or are governed by another path).
# =============================================================================
# audit-sink-canonicality.mjs FAILS when a STDLIB_CAPABILITY_MAP entry whose name/shape
# denotes an egress / exec / filesystem-write / off-host-send sink is present in NONE of
# the keyed sink hand-lists (taint-checker INJECTION_SINKS, value-state SINK_REQUIREMENTS
# exact keys, source-escape BANNED_CALL_NAMES). That is the RD-0234 drift class: a
# newly-added stdlib sink silently escapes taint / value-state tracking, so a tainted or
# raw-secret value reaches it with no diagnostic and the artifact signs clean.
#
# The shape classifier is deliberately broad (better to over-flag than miss a sink). An
# entry that is sink-SHAPED but is genuinely tracked by a DIFFERENT enforcement path than
# the three keyed hand-lists — value-state's REGEX-pattern sinks or its receiver-segment
# secret-egress guard (isNetworkSink) — is recorded HERE with the specific citation, so
# the coverage knowledge stays auditable in one place.
#
# Format:  <StdlibName>   # concrete reason it need not appear in a keyed sink list
# Add an entry ONLY after running `node scripts/audit-sink-canonicality.mjs` and
# confirming (ideally by compiling a probe fixture) how the entry is governed.
# =============================================================================

#    (non-secret) coverage flows through the lowercase /^https?\./ pattern above; the
#    capitalised spelling is caught for secret egress. Recognised, not a silent escape. ─
